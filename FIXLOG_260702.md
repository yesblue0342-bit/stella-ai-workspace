## 근본 원인 (코드 레벨 확정 — 추측 아님)

### 1) 첨부 파일 내용 미반영 ("양식을 알지 못해")
- **증거**: 실사용 로그에서 첨부 직후 턴은 실제 데이터(Celltrion Branchburg, ZAQMR0060 등) 반영
  → **후속 턴**("아니 첨부한 양식대로")에서만 "양식을 알지 못해". 파서(mammoth/xlsx/pdf.js/OCR)는 정상.
- **원인**: `index.html send()` — 첨부 텍스트(fileText)가 **현재 턴 message에만** 합쳐지고,
  대화 저장(`r.messages.push({role,text})`)과 히스토리(`map(m=>m.text)`)에는 **빠짐**.
  → 다음 턴부터 AI가 첨부 내용을 완전히 잊음.
- **수정**: 메시지 저장 시 `att`(파일명+텍스트, 파일당 6000자·최대 3개 캡) 보존 +
  신설 `buildChatHistory()`가 히스토리에 첨부 텍스트 주입(4000자 캡, **마지막=현재 메시지는
  제외**해 현재 턴 전문 전달과 중복 방지).

### 2) 가짜 파일 생성 응답 ("준비했습니다, 다운로드하세요")
- **원인 2중**: ① 모델의 거짓 비동기 약속(파일 생성 능력 없음) — 직전 배포(ebcf467)에서
  전 경로 프롬프트로 금지 완료. ② 다운로드 버튼이 구버전 캐시로 미표시 — 직전 배포에서
  새 SW 자동 감지·자동 새로고침 추가 완료.
- **이번 보강**: Word/PPT 다운로드가 원문 마크다운 텍스트가 아니라 **렌더된 HTML**
  (`answerHtmlForExport`)로 생성 → Word에서 진짜 표(테두리)·제목 서식으로 열림.
  Malgun Gothic 지정으로 한글 안전.

### 3) Google Drive 링크 참조 실패 (사용자 추정 → 코드로 확정)
- **원인**: `detectDrivePathText`는 `#폴더`/`내 드라이브 >` 패턴만 인식.
  `drive.google.com/...` URL은 needsDrive는 발동하나 경로 해석 실패 →
  "경로를 인식하지 못했습니다"로 종료. `docs.google.com`은 needsDrive조차 미발동.
- **수정**: `detectDriveLink()` 신설(`file/d/ID`, `open?id`, `uc?id`, `/folders/ID`,
  docs.google.com 문서/시트/프레젠테이션) → `buildDriveContextForChat`가 링크 ID로
  `readDriveTarget` 직접 호출. 링크 읽기 실패 시 무의미한 키워드 폴백 대신 정직한
  권한 안내. `chat.js needsDrive`에 docs.google.com 추가.
